# Dockerfile 打包镜像 + 外部配置挂载 完整流程

## 流程概览

```
编写 Dockerfile，定义镜像构建规则
    ↓
docker build 构建镜像
    ↓
修改 application.properties，把 localhost 改成宿主机 IP
    ↓
用 -v 挂载外部配置文件，启动容器
    ↓
验证容器内应用正常运行
```

---

## 第一步：编写 Dockerfile

在项目根目录创建 `Dockerfile` 文件（与 `pom.xml` 同级）：

```dockerfile
FROM amazoncorretto:8-alpine

RUN mkdir /app

WORKDIR /app

COPY target/xdml-v1-0.1.0.jar /app

EXPOSE 8080

CMD [ "java", "-jar", "xdml-v1-0.1.0.jar" ]
```

**每行说明：**

| 指令 | 作用 |
|------|------|
| `FROM amazoncorretto:8-alpine` | 基础镜像，包含 JDK 8，alpine 是精简版 Linux，体积小。**注意：`java:openjdk-8u111-alpine` 是 x86 架构，在 Apple Silicon（M系列）Mac 上无法运行，需改用 `amazoncorretto:8-alpine`（原生支持 arm64）** |
| `RUN mkdir /app` | 在容器内创建 `/app` 目录 |
| `WORKDIR /app` | 设置工作目录，后续命令都在 `/app` 下执行 |
| `COPY target/xdml-v1-0.1.0.jar /app` | 把本机打好的 jar 复制进镜像 |
| `EXPOSE 8080` | 声明容器对外暴露 8080 端口（仅文档作用，实际映射需 `-p`） |
| `CMD [...]` | 容器启动时执行的命令，即 `java -jar xdml-v1-0.1.0.jar` |

> 注意：执行 `docker build` 之前需要先执行 `mvn package` 确保 `target/` 下有 jar 文件。

---

## 第二步：构建 Docker 镜像

```bash
docker build .
```

执行后 Docker 会逐步执行 Dockerfile 里的每条指令：

```
Sending build context to Docker daemon  35.94MB
Step 1/6 : FROM java:openjdk-8u111-alpine
Step 2/6 : RUN mkdir /app
Step 3/6 : WORKDIR /app
Step 4/6 : COPY target/xdml-v1-0.1.0.jar /app
Step 5/6 : EXPOSE 8080
Step 6/6 : CMD [ "java", "-jar", "xdml-v1-0.1.0.jar" ]
```

**作用：** 把应用及其运行环境打包成一个镜像，任何安装了 Docker 的机器都可以直接运行这个镜像，不需要单独安装 JDK。

构建完成后查看镜像，并给镜像打 tag：

```bash
docker images
```

刚构建的镜像 REPOSITORY 和 TAG 都是 `<none>`，用 `docker tag` 给它命名：

```bash
docker tag <IMAGE_ID> xdml-v1-build:0.1.0
```

- `xdml-v1-build`：对应项目目录名
- `0.1.0`：对应 `pom.xml` 里的版本号

如果需要重命名，先打新 tag 再删旧 tag：

```bash
docker tag xdml-v1:0.1.0 xdml-v1-build:0.1.0
docker rmi xdml-v1:0.1.0
```

命名后启动容器可以用名字代替 ID：

```bash
docker run -v `pwd`/application.properties:/app/config/application.properties -p 8082:8080 xdml-v1-build:0.1.0
```

## 第三步：修改 application.properties，把 localhost 改成宿主机 IP

先查本机 IP：

```bash
ifconfig
# 找 en0 下的 inet，例如 192.168.31.250
```

然后编辑配置文件：

```bash
vi application.properties
```

把所有 `localhost` 改成宿主机局域网 IP：

```properties
spring.datasource.url=jdbc:mysql://192.168.31.250:3306/match
spring.datasource.username=root
spring.datasource.password=root
spring.datasource.driver-class-name=com.mysql.cj.jdbc.Driver
mybatis.config-location=classpath:db/mybatis/config.xml
spring.aop.proxy-target-class=true

spring.redis.host=192.168.31.250
# Redis服务器连接端口
spring.redis.port=6379
```

> **注意：** `mybatis.config-location` 的文件名必须和 jar 包内实际的文件名一致。可以用以下命令确认 jar 包内的实际路径：
> ```bash
> jar tf target/xdml-v1-0.1.0.jar | grep mybatis
> ```

**作用：** Docker 容器有独立的网络命名空间，容器内的 `localhost` 指的是容器自身，无法访问宿主机上的 MySQL 和 Redis。必须改成宿主机的局域网 IP，容器才能连上这两个服务。

## 第四步：用 -v 挂载外部配置文件启动容器

```bash
docker run -v `pwd`/application.properties:/app/config/application.properties -p 8082:8080 <镜像ID>
```

命名后启动容器可以用名字代替 ID：

```bash
docker run -v `pwd`/application.properties:/app/config/application.properties -p 8082:8080 xdml-v1-build:0.1.0
```


**参数说明：**
- `-v \`pwd\`/application.properties:/app/config/application.properties`：把当前目录的 `application.properties` 挂载到容器内的 `/app/config/` 目录下
- `-p 8082:8080`：把容器内的 8080 端口映射到宿主机的 8082 端口

**为什么挂载到 `/app/config/`：**

根据 Spring Boot 官方文档，`application.properties` 的加载优先级从高到低为：

```
1. 当前目录的 /config 子目录     ← 最高优先级
2. 当前目录
3. classpath 的 /config 包
4. classpath 根目录              ← 最低优先级
```

容器的工作目录是 `/app`（`WORKDIR /app`），所以把配置挂载到 `/app/config/application.properties`，Spring Boot 会优先读取这份外部配置，**覆盖** jar 包内打包的配置文件。

**作用：** 不需要重新构建镜像就能修改配置。把配置文件放在容器外部挂载进去，修改配置只需改文件再重启容器，镜像本身不变。

## 第五步：验证容器正常运行

```bash
docker ps
```

确认容器在运行后，给容器改个便于管理的名字：

```bash
docker rename <原容器名> my-xdml-build-dock
```

访问应用：

```bash
curl http://localhost:8082
# 或浏览器访问 http://localhost:8082
```

**查看实时日志，验证请求是否打到容器：**

```bash
docker logs -f my-xdml
```

`-f` 表示持续跟踪日志（follow）。刷新浏览器页面，终端里每次都会打印：

```
Get request!
```

看到这行日志说明请求成功进入了容器内的应用，负载均衡也在正常工作。

如果 Nginx 已配置好 upstream，也可以通过域名访问：

```
http://abc.com
```

## Spring Boot 配置文件加载优先级（补充）

| 优先级 | 位置 |
|--------|------|
| 1（最高） | 当前工作目录的 `/config` 子目录 |
| 2 | 当前工作目录 |
| 3 | classpath 下的 `/config` 包 |
| 4（最低） | classpath 根目录（即 jar 包内的 `application.properties`） |

优先级高的配置会覆盖优先级低的配置，所以挂载到 `/app/config/` 的外部配置文件会覆盖 jar 包里的默认配置。

## 常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| `docker build` 报找不到 jar | 未打包或路径不对 | 先执行 `mvn package`，确认 `target/` 下有 jar |
| 容器启动后连不上 MySQL/Redis | 配置里用了 `localhost` | 改成宿主机局域网 IP（`ifconfig` 查） |
| 修改了配置但没生效 | 容器未重启 | `docker restart <容器ID>` |
| 外部配置未覆盖 jar 内配置 | 挂载路径不对 | 确认挂载到 `WORKDIR/config/application.properties` |
| 容器启动后立刻退出，日志为空 | 基础镜像架构不匹配（x86 镜像在 arm64 上无法运行） | Dockerfile 改用 `amazoncorretto:8-alpine` 重新构建 |
| `FileNotFoundException: db/mybatis/config.xml` | 外部配置的 `mybatis.config-location` 文件名与 jar 包内不一致 | 用 `jar tf target/*.jar \| grep mybatis` 确认实际文件名 |
